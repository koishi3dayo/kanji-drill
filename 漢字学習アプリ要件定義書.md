# 漢字学習アプリ 要件定義書

- **プロジェクト名**：漢字ドリル（小学生向け 漢字学習アプリ）
- **作成日**：2026-07-03
- **バージョン**：v2.2（仕様変更反映版）
- **開発手法**：AI-DLC（AI-Driven Life Cycle）

---

## 1. 背景・目的

小学2〜5年生（特に4〜5年生重点）の漢字定着を、**毎日10問のドリル**で習慣化する。
学習の継続と達成感を、進捗の可視化（ヒートマップ）とゲーミフィケーション（連続日数・バッジ）で支える。

---

## 2. 仕様変更の記録

### 2.1 変更の経緯

| 版 | 入力・採点方式 | 課題 |
|----|--------------|------|
| v1.0（当初） | PCのキーボード＋IMEで漢字を入力して自動採点 | IMEが漢字を候補表示するため「思い出して書く」練習にならない。学習効果が薄い |
| v1.5（検討） | タブレット/スマホで手書きキャンバス入力 | 手書き対応端末が必要。本PCは非対応 |
| **v2.0（本仕様・確定）** | **PC出題 → 紙に手書き回答 → PCで答え合わせ・自己採点** | 課題を解消。本物の書き練習は紙で行う |

### 2.2 v2.0 の基本方針（確定仕様）

1. **電子端末への手書き入力は行わない。**
2. **PCが問題を出題する。**（読み・熟語・用例のヒントを画面表示）
3. **児童は問題を見て、紙に漢字を手書きで回答する。**
4. **PCで正解を表示し、児童本人または保護者が答え合わせをする。**
5. **正解／不正解をPC上でチェックボックス（○×）で記録する。**
6. **記録の履歴から、学習進捗と弱点をヒートマップで可視化する。**

## 4. 出題仕様

### 4.1 出題数・頻度
- **1日あたり10問**（毎日実施を想定）。
- 1日1回のセッションを基本とする（再挑戦は可）。
- **（v2.2）10問すべてを最初から画面に一覧表示する。**1問ずつの逐次出題は行わない。

### 4.1.1 出題学年の重み付け（v2.2 新規）
- **出題する漢字は5年生を中心とする。**
- **10問中9問は5年生の配当漢字から出題する。**
- 残り1問は、前日の不正解の復習や他学年（2〜4年生）の定着確認にあてる。
- 前日不正解の復習を優先する際も、この「9（5年生）：1（他学年）」の比率は維持する。

### 4.2 問題タイプ（v2.1 改訂）
| タイプ | 出題内容（画面表示） | 紙の回答 |
|--------|---------------------|---------|
| 書き（訓読み中心） | かなの短文を提示（下線部が対象） | 下線部を漢字で書く |
| 熟語（音読み中心） | かなの短文を提示（下線部が対象） | 下線部の熟語を漢字で書く |

- 出題はすべて**5〜10文字程度のかな短文**とする。単語だけの出題はしない。
  - NG：「その」→ 園
  - OK：「うたをうたう」→ 歌を歌う ／「こうえんであそぶ」→ 公園であそぶ
- 短文データは `sentence-data.js` に全漢字ぶん収録（かな文・漢字文・タイプ）。
- 音読み・訓読み・熟語をバランスよく混在させる。
- **（v2.2）問題文中、出題対象（答えさせる箇所）以外の部分は、極力すでに習った漢字に変換して表示する。**
  - 例：「となりのへやにいどう」（全部かな）→「となりの部屋に移動」（対象の「いどう」以外はすでに習った熟語「部屋」で表示）
  - 全文かな表記は避け、実際の文章に近い読みごたえのある問題にする。
  - データ形式を `{tpl, reading, type}` に刷新（`tpl` は `{{対象}}` マーカーで出題箇所を示す既習漢字まじりテンプレート）。
  - **実装方法**：`kanji-data.js` の熟語（2文字以上の単語＋読み）を辞書化し、既存の短文データへ機械的に置換するスクリプト（`scripts/convert_sentences.py`）で全550件超を一括変換。
    - 単漢字の訓読み（1文字の読み）は助詞（の・に・を等）と同音衝突し誤変換を起こすため、**辞書化の対象外**とした（例：「の」が「野」に誤変換される事故を確認し除外）。
    - 変換後、辞書内の熟語どうしの同音異義語衝突（例：「こうじょう」＝向上／工場）を全件突合検証し、誤った語が採用されていないことを確認済み。

### 4.2.1 用語の注記
- 「用例」タイプは v2.2 にて「書き」「熟語」の2タイプに整理・統合済み（4.2参照）。

### 4.3 出題ロジック（適応的出題）
1. **前日に不正解だった問題を優先**して当日の10問に含める（最大5問）。
2. 残りは**正解率の低い漢字・未出題の漢字を優先**して選出。
3. 問題タイプ（書き・熟語・用例）をバランスよく混ぜる。
4. 出題順はシャッフルする。

## 5. 採点・記録仕様（v2.0 中核）

### 5.1 答え合わせフロー（v2.1 改訂：一括答え合わせ方式）
```
1. PC画面に第1問〜第10問を連続で出題（答えは見せない）
2. 児童は紙に①〜⑩の番号をつけて漢字を書いていく
3. 10問終了後、「答えあわせ」画面で全10問の正解を一覧表示
4. 児童本人 or 保護者が紙と正解を照合
5. 各問題の ⭕（正解）／❌（不正解）を選んで記録
6. 全問チェック後「きろくする」で保存 → 結果画面へ
```
- 出題中は「まえへ」ボタンで前の問題に戻れる（テスト中の見直し用）。
- ⭕❌を10問すべて選ぶまで記録ボタンは押せない（未採点防止）。

### 5.2 記録項目
| 項目 | 内容 |
|------|------|
| 日付 | 実施日（YYYY-MM-DD） |
| 漢字＋学年 | 出題された漢字と配当学年 |
| 問題タイプ | 書き／熟語／用例 |
| 正誤 | ○（正解）／×（不正解）※本人・保護者が入力 |

- 記録は端末内（localStorage）に保存し、外部送信しない。

## 6. 可視化仕様

### 6.1 ヒートマップ（進捗マップ）
- 学年タブ（2〜5年生）で切り替え。
- 各漢字を正解率で色分け：
  | 色 | 意味 |
  |----|------|
  | グレー | 未出題 |
  | 赤 | 正解率 低（〜50%未満） |
  | 黄 | 正解率 中（50〜80%） |
  | 緑 | 正解率 高（80〜100%未満） |
  | 濃緑 | 完璧（100%） |
- セルタップで、その漢字の読み・熟語・成績を表示。

### 6.2 進捗・達成感の演出
| 要素 | 内容 |
|------|------|
| 連続日数（ストリーク） | 毎日の継続を🔥で表示 |
| バッジ | 3日/7日/30日/100日連続、累計問題数で獲得 |
| 週間成績 | 直近7日の解答数・正解数・正解率 |
| 完了バナー | 当日のテスト完了を明示し達成感を付与 |

### 6.3 学習カレンダー（v2.2 新規）
- 月表示のカレンダーを表示する。
- **学習を実施した日には自動でチェックマーク（✅）を付ける。**（記録保存＝実施日として即反映）
- 月の切り替え（前月・翌月）ができる。
- チェック済みの日をタップすると、その日のスコア（例：8/10）を確認できる。
- 目的：継続の可視化による達成感の強化（ヒートマップが「どの漢字が弱いか」、カレンダーは「いつ頑張ったか」を可視化する）。

## 8.5 タブレット/スマホ対応（v2.3　A・B実装済み／C公開は保留）

### 8.5.1 方針（確定）
- **配信方法：無料の静的ホスティング（GitHub Pages）＋ PWA化。**
  - URLでどこからでもアクセス可能。PCがオフでも使える。
  - ホーム画面にアイコンを追加し、ブラウザのURLバーなしで全画面起動。オフラインでも起動可能。
- **データ共有：行わない（1台運用）。** 学習記録はその端末内（localStorage）のみ。端末をまたいだ同期はしない。
  - → 「学習データは端末内のみ・外部送信なし」のプライバシー方針を維持できる。

### 8.5.2 変更内容

**A. レスポンシブ／タッチ対応（CSS中心）**
- タップ領域を最低44px確保（⭕❌ボタン・カレンダー日付・ヒートマップセル・タブ）。
- iOS Safari対策：`viewport-fit=cover` ＋ `safe-area-inset`（ノッチ/ホームバー回避）、`100dvh` 採用、ダブルタップ拡大の抑制（アクセシビリティのピンチズームは残す）。
- 狭い画面（〜360px）向けのフォント・余白の調整、一覧グリッドの列数自動化。

**B. PWA化**
| ファイル | 役割 |
|---------|------|
| `manifest.json` | アプリ名・アイコン・全画面表示（standalone）・テーマ色・縦向き固定 |
| アイコン（192/512px, maskable） | ホーム画面アイコン。iOS用 `apple-touch-icon` も設定 |
| `sw.js`（Service Worker） | `index.html`・データ・アイコンをキャッシュし**オフライン起動**を実現 |
- 更新時の対策：Service Workerのキャッシュにバージョンを持たせ、新版公開時に自動更新する。

**C. 配信（GitHub Pages）**
- `kanji-drill` をGitリポジトリ化し、GitHubへpush → Pagesを有効化。
- 公開URL例：`https://koishi3dayo.github.io/kanji-drill/`
- **注意**：公開されるのはアプリのコード（漢字ドリル本体）のみ。**個人情報・学習記録は含まれない**（記録は各端末のlocalStorageに留まるため）。ただしURLを知れば誰でもアプリを開ける状態にはなる。

### 8.5.3 動作確認方針
- プレビューを 375px（スマホ）・768px（タブレット）で検証。
- タッチ操作（⭕❌・タブ・カレンダー）・全画面表示・オフライン起動を確認。

### 8.5.4 実装状況（2026-07-04 時点）
- **A（レスポンシブ／タッチ）実装・検証済み**：375px/768pxで表示確認。safe-area・44pxタップ領域・狭幅調整を適用。
- **B（PWA化）実装・検証済み**：`manifest.json`・`sw.js`・アイコン（192/512/512-maskable/180）を追加。SW登録・全アセットのキャッシュ（オフライン起動可能）を確認。生成スクリプトは `scripts/make_icons.py`。
- **C（GitHub Pages公開）は保留中**：外部公開のため、別途タイミングを相談して実施する。
